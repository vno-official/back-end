## ⚙️ Restructuring Rules

### 1. Root Configuration
- The root `settings.gradle` must declare all subprojects:
  ```groovy
  rootProject.name = 'vno-backend'
  include(
      'common',
      'contracts',
      'config-server',
      'service-discovery',
      'api-gateway',
      'auth-service',
      'user-service',
      'organization-service',
      'note-service',
      'tag-service',
      'notification-service'
  )
  ```

- The root `build.gradle` must define global dependency management and plugin versions.

### 2. Common Module Rules
- Must **not depend on Spring Boot**.
- Use `java-library` plugin.
- Allowed dependencies:
  - `lombok`
  - `jackson-databind`
  - `jakarta.validation-api`
  - `slf4j-api`
- Contains:
  - Shared DTOs, Enums
  - Exception models & handlers
  - JWT utility (pure Java)
  - Logging and date utilities

### 3. Contracts Module (NEW - rules & usage)
- Purpose: **single source of truth** for API definitions used by backend & frontend.
- Subfolders: `openapi/`, `proto/`, `asyncapi/`.
- Rules:
  - **Contract-first** approach: update contracts first, then generate server/client stubs.
  - Use semantic versioning for contract files (e.g. `auth-service.v1.yaml`, `user-service.v2.yaml`) or a changelog file per contract.
  - Keep `gateway-aggregated.yaml` generated by CI from service OpenAPI outputs.
  - For Protobufs:
    - Keep `.proto` under `contracts/proto/` and generate Java classes into each service where needed.
    - Add protoc generation step in Gradle build.
  - For AsyncAPI (events):
    - Define event schemas and channels here; consumers/producers must reference these file versions.
- CI Enforcement:
  - Lint contracts (OpenAPI linter, proto lint).
  - Run contract compatibility checks on PR (prevent breaking changes without a major version bump).
  - Auto-generate server stubs and client SDKs when contracts change.

### 4. Config Server
- Use `spring-cloud-config-server`
- Loads configuration from `config-repo/`
- Expose `/actuator/refresh` for runtime updates

### 5. Service Discovery
- Use `spring-cloud-starter-netflix-eureka-server`
- Register all services dynamically

### 6. API Gateway
- Use `spring-cloud-gateway`
- Responsibilities:
  - Validate JWT before routing
  - Forward only valid requests
  - Load routes from `application.yml` or config-server
- Integrate `springdoc-openapi-starter-gateway` for aggregated API docs
- Gateway should host the unified Swagger UI that points to `gateway-aggregated.yaml` produced by CI.

### 7. Business Services (auth, user, note, etc.)
- Each service should:
  - Register with Eureka
  - Fetch configuration from config-server
  - Expose REST endpoints with `springdoc-openapi`
  - Use Feign clients for cross-service calls
- Use `common` module for shared DTOs & utilities

### 8. Security Rules
- JWT validation happens only at Gateway
- Services trust Gateway or internal service token
- Auth-service issues and refreshes JWT tokens
- Propagate identity/context headers (e.g., `X-User-Id`, `X-Request-Id`) to downstream services

### 9. Documentation
- Each service includes:
  ```groovy
  implementation 'org.springdoc:springdoc-openapi-starter-webmvc-ui:2.5.0'
  ```
- Define `OpenApiConfig.java` with title, version, description
- CI should produce `gateway-aggregated.yaml` by merging all service OpenAPI specs
- Frontend should point to Gateway Swagger UI
- Maintain a human-readable CHANGELOG for contracts

### 10. Build & Deployment
- Use Dockerfile per service
- Root `docker-compose.yml` runs the full stack in correct order:
  1. Config Server  
  2. Eureka Server  
  3. Auth Service  
  4. Core Services  
  5. API Gateway  

### 11. Maintainability & Scalability
- Enforce clean boundaries:
  - No cross-service direct imports
  - Communication only via REST/Feign/gRPC/messages
- Enable metrics (Micrometer, Zipkin, Prometheus)
- Add CI/CD pipeline for Gradle build & container deployment
- Keep `common` minimal and reusable

## 🔍 Contracts — Short Analysis & Best Practices (what you were missing)
1. **Why contracts matter**: They are the canonical API spec; frontend and backend must agree on them. Centralizing them avoids drift.
2. **Contract-first workflow**:
   - Update contract in `contracts/` → Commit → CI validates → Generate server/client stubs → Run integration tests.
3. **Versioning & compatibility**:
   - Use semantic versioning per contract file.
   - Add compatibility checks (e.g., openapi-diff or swagger-diff) to CI to prevent accidental breaking changes.
4. **Aggregated API docs**:
   - CI job to merge service OpenAPI docs into `gateway-aggregated.yaml` and publish to gateway Swagger UI.
5. **Proto & gRPC**:
   - Store `.proto` in `contracts/proto/` and use `protoc` to generate code during build.
   - Treat proto as strict contract: breaking changes must bump major version.
6. **Event-driven contracts (AsyncAPI)**:
   - Define event payload schemas and channel names; version events; consumers must subscribe to specific versions.
7. **Tooling recommendations**:
   - OpenAPI/Swagger Editor for editing
   - `swagger-cli` / `openapi-cli` for linting & bundling
   - `buf` for protobuf linting and breaking change detection
   - `asyncapi` tools for event contract linting
8. **Automation**:
   - Auto-generate client SDKs for FE (TypeScript) and publish to an internal npm registry if needed.
   - Gate PRs with contract tests and compatibility checks.
